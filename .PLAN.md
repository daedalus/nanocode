# nanocode: Integration Plan (Aura-IDE + MiMo-Code)

## Quick Summary

| Source | Focus Areas | Key Features |
|--------|-------------|--------------|
| **Aura-IDE** | Code editing, tool system, architecture | 4-tier edit matching, AST symbol editing, BM25 codebase index, planner/worker dispatch |
| **MiMo-Code** | Memory, tasks, context management | SQLite FTS5 memory search, tree-shaped tasks, context reconstruction, skill discovery |

**Total estimated effort:** ~80-100 hours across both sources

## Source Repositories

- **Aura-IDE**: `~/code/Aura-IDE` - Mature Qt-based AI coding harness
- **MiMo-Code**: `~/code/MiMo-Code` - Open-source AI coding agent with cross-session memory

## Overview

This document catalogs features from two mature AI coding agents that nanocode should adopt, prioritized by impact vs effort:

1. **Aura-IDE** (`~/code/Aura-IDE`): Mature Qt-based AI coding harness (1.4.4) with ~3-4x more feature-complete non-GUI layer
2. **MiMo-Code** (`~/code/MiMo-Code`): Open-source AI coding agent with advanced memory, task management, and context handling

**Current nanocode strength:** Event-based streaming pipeline, MCP/LSP integration, multi-agent system, snapshot git, context chips, planning, storage/SQLite, skills system, hooks, drift detection, todos, retry/eBPF guard, doom loop detection.

**Aura strengths nanocode lacks:** Planner/Worker dispatch, AST-based symbol editing, fuzzy edit matching, BM25 codebase index, hardware-tethered key encryption, recovery state machine, pre-write backups, composable persona prompts, focused actions, edit transactions, project memory DB, dual-engine grep, tool parallelism, cost tracking, provider registry, docker sandbox, vision preprocessing, research sub-agent.

**MiMo-Code strengths nanocode lacks:** Persistent memory with SQLite FTS5 search, tree-shaped task system, intelligent context reconstruction, skill discovery, dream/distill self-improvement, goal/stop conditions.

## Priority 1: High Impact, Low Effort (Quick Wins)

### 1.1 4-Tier Edit Matching (fs_write.py: `propose_edit`) ✅

**Source:** `aura/conversation/tools/fs_write.py`

**What:** Replace nanocode's simple `str.replace()` edit strategy with Aura's 4-tier matching:
1. Exact string match
2. Line-by-line exact match (split old_str into lines)
3. Whitespace-agnostic fuzzy matching (difflib.SequenceMatcher, threshold 0.75)
4. Failure with nearest candidates

Also: auto-strip markdown fences (models always wrap code in ```), syntax validation via `compile()` for .py files.

**Effort:** ~2 hours. Drop-in replacement for `EditTool.execute()` in `tools/builtin/__init__.py`.

**Files to create/modify:**
- `nanocode/tools/builtin/edit.py` (new module, extract from EditTool)
- `nanocode/tools/builtin/__init__.py` (wire it in)

### 1.2 AST-Based Symbol Editing (fs_edit_structured.py: `edit_symbol`) ✅

**Source:** `aura/conversation/tools/fs_edit_structured.py`

**What:** Add `edit_symbol` tool that replaces Python functions/classes/methods by name via `ast` module. Completely bypasses string matching hell. Handles decorators, auto-indents replacement, validates output.

**Effort:** ~2 hours.

**Files:**
- `nanocode/tools/builtin/edit_symbol.py`

### 1.3 Markdown Fence Sanitization in Edit ✅

**Source:** `aura/conversation/tools/fs_write.py` (fence sanitization logic)

**What:** Before attempting any edit match, strip ```python and ``` fences from old_str. This eliminates the #1 failure mode for LLM-generated edits.

**Effort:** ~30 min, inline in EditTool.

### 1.4 Find Usages Tool (find_usages.py)

**Source:** `aura/conversation/tools/find_usages.py`

**What:** Simple tool wrapping grep with `\b{symbol}\b` word-boundary matching. Helps models find where a symbol is used without regex knowledge.

**Effort:** ~30 min.

**Files:**
- `nanocode/tools/builtin/find_usages.py`

### 1.5 Pre-Write Backups (backup.py)

**Source:** `aura/conversation/tools/backup.py`

**What:** Before any write_file/edit_file, copy existing file to `.nanocode/backups/<ISO-timestamp>/<relpath>`. Simple, timestamped, reversible. nanocode already has `snapshot/git.py` for git-level snapshots but lacks file-level pre-write backups.

**Effort:** ~1 hour.

**Files:**
- `nanocode/tools/backup.py`
- Modify `WriteFileTool` and `EditTool` to call `backup_existing()` before write.

### 1.6 Negative Prompting Blocks (prompts.py)

**Source:** `aura/prompts.py` (`_CODE_TASTE_BLOCK`, `_ARCHITECTURE_GUARDRAILS`, `_CODE_QUALITY_CONTRACT`, `_CODE_STYLE_EXAMPLES`, `_WORKER_PASS_RULES`)

**What:** Add composable persona prompt blocks that explicitly describe what NOT to do. Aura's negative prompting ("no tutorial docstrings", "no manager-of-managers pattern", "no generic names") is remarkably effective.

**Effort:** ~1 hour.

**Files:**
- `nanocode/prompts/blocks/` (new dir)
- Modify `nanocode/core.py` `_build_system_prompt()` to compose blocks.

## Priority 2: High Impact, Medium Effort

### 2.1 BM25 Codebase Index (codebase_index/) ✅

**Source:** `aura/codebase_index/bm25.py`, `indexer.py`, `tool.py`

**What:** Pure-Python BM25 inverted index over workspace files with:
- Smart code-aware tokenizer (camelCase + snake_case)
- Incremental refresh via mtime
- Disk cache (SHA-256 workspace hash key)
- 1500 file / 128KB per file limit
- `search_codebase` tool that ranks files by semantic relevance

This is nanocode's biggest gap — current grep-only search is far worse for "find where X is defined/used."

**Effort:** ~4 hours to port, ~2 to integrate.

**Files:**
- `nanocode/codebase_index/__init__.py`
- `nanocode/codebase_index/bm25.py`
- `nanocode/codebase_index/indexer.py`
- `nanocode/codebase_index/tool.py`

### 2.2 Hardware-Tethered Key Manager (key_manager.py)

**Source:** `aura/key_manager.py`

**What:** Fallback for when OS keyring is unavailable. Machine key derived from `uuid.getnode()` + `os.getlogin()` → SHA-256 → base64 Fernet key. Stores encrypted keys at `~/.config/nanocode/keys.json` with `0o600` permissions. Auto-migrates legacy plaintext keys.

nanocode's current `keyring.py` depends on `keyring` library which is often unavailable in headless/server environments.

**Effort:** ~2 hours.

**Files:**
- `nanocode/key_manager.py`
- Modify `keyring.py` or merge into `EnvKeyringManager`.

### 2.3 Recovery State Machine (conversation/manager.py)

**Source:** `aura/conversation/manager.py` (ToolLimitState, edit_failed_shapes, syntax_repair_required, compiler_repair_required)

**What:** Track per-turn: which edit patterns failed, syntax repair state, write attempts per path. Inject recovery nudges when the model repeats failures — instead of just hitting a limit and dying.

nanocode has `doom_loop.py` (detects repeated identical calls) but lacks Aura's richer failure tracking.

**Effort:** ~3 hours.

**Files:**
- `nanocode/recovery.py` (new module)
- Integrate into `core.py` `_handle_tool_calls()`.

### 2.4 AST-Based Repo Map (repo_map.py)

**Source:** `aura/repo_map.py`

**What:** Structural workspace overview injected into system prompt: class names, base classes, method signatures, top-level functions extracted via `ast`. Capped at 300 lines, mtime-cached, 95%+ prompt cache hit rate.

nanocode has context chips but no structured repo map. This is especially valuable for prompt caching.

**Effort:** ~2 hours.

**Files:**
- `nanocode/repo_map.py` ✅
- Integrate into `_build_system_prompt()` in `core.py`. ✅

### 2.5 Git Introspection Tools (git_tools.py)

**Source:** `aura/conversation/tools/git_tools.py`

**What:** Add `git_log_file` (follow renames), `git_branch_list` (local + tracking), `git_stash_list`, `git_stash_show`. nanocode's `snapshot/git.py` is focused on snapshot/restore, not rich introspection.

**Effort:** ~2 hours.

**Files:**
- `nanocode/tools/builtin/git_tools.py`

## Priority 3: High Impact, Higher Effort

### 3.1 Planner/Worker Dispatch (dispatch.py, conversation/manager.py)

**Source:** `aura/conversation/dispatch.py`, `aura/conversation/manager.py`

**What:** Two-agent architecture: Planner (read-only + explore → writes spec) → Worker (reads spec, executes, validates). nanocode's planning system (`planning/`) is more about task decomposition; Aura's is about role separation with different models/temperatures/thinking modes per role.

Key components to port:
- `WorkerDispatchRequest` / `WorkerDispatchResult` dataclasses
- `ExplicitSpecContract` (expected symbols, forbidden methods)
- `WorkerOutcomeStatus` enum (13 states)
- `dispatch_to_worker` tool (Planner-only)

**Effort:** ~8-12 hours for full implementation.

### 3.2 Edit Transactions (fs_edit_transaction.py)

**Source:** `aura/conversation/tools/fs_edit_transaction.py`

**What:** Atomic multi-operation edits with content-hash race-condition safety. Operations: `replace_text_once`, `replace_text_all`, `insert_before`, `insert_after`, `delete_lines`, `move_lines`. Each op references a unique content hash for safety.

**Effort:** ~4 hours.

### 3.3 Dual-Engine Grep (grep.py)

**Source:** `aura/conversation/tools/grep.py`

**What:** Try ripgrep first (much faster), fall back to pure-Python regex search. Auto-retry as regex if literal search fails and pattern looks regex-like.

**Effort:** ~2 hours.

### 3.4 Memory DB (SQLite+FTS5) (memory_db.py)

**Source:** `aura/memory_db.py`

**What:** Dedicated project memory database with SQLite+FTS5 full-text search. Auto-saves completed Worker dispatch records. Both AI (via `search_project_memory` tool) and user (via info panel) can query.

nanocode has SQLite storage but no dedicated memory/retrieval layer with FTS5.

**Effort:** ~3 hours.

### 3.5 Typed Streaming Events Enhancement

**Source:** `aura/client/events.py`

**What:** nanocode already has `llm/events.py` with typed events, but Aura's are more refined:
- `ToolCallStart`/`ToolCallArgsDelta`/`ToolCallEnd` pattern (nanocode has single `ToolCallEvent`)
- `Usage` event with cache_hit/cache_miss buckets
- `ApiError` as a first-class event (nanocode throws exceptions)
- `WorkerDispatchRequested` event for planner/worker handoff

**Effort:** ~2 hours.

### 3.6 Tool Parallelism for Reads ✅

**Source:** `aura/conversation/manager.py` (ThreadPoolExecutor for read-only tools)

**What:** Execute read-only tools (read, grep, glob, find_usages) concurrently via ThreadPoolExecutor. Write tools remain sequential.

**Implementation:**
1. ✅ Created ToolParallelismManager class
2. ✅ Defined read/write tool classifications
3. ✅ Integrated with ToolExecutor
4. ✅ Added semaphore-based concurrency control

**Effort:** ~2 hours

**Files created:**
- `nanocode/tools/parallel.py` - ToolParallelismManager with read/write classification
- `tests/test_tool_parallel.py` - 18 unit tests (all passing)

### 3.7 Cost Tracking

**Source:** `aura/config.py` (pricing tables), `aura/client/events.py` (Usage event)

**What:** Per-model pricing with three buckets: `input_miss`, `input_hit`, `output`. Track session cost, display in TUI/CLI. Aura already has embedded pricing tables for all major models.

**Effort:** ~3 hours.

## Priority 4: Nice-to-Have

### 4.1 Provider Registry Pattern

**Source:** `aura/providers/registry.py`, `aura/providers/base.py`

**What:** Each provider is a `ProviderSpec` with `ModelInfo` per model. Dynamic catalog fetching from OpenRouter cached to disk. Much cleaner separation than nanocode's flat config dict.

### 4.2 Dynamic Tools (.nanocode/tools/)

**Source:** `aura/conversation/tools/dynamic.py`

**What:** AST-parse `.nanocode/tools/*.py` files, extract function signatures as OpenAI tool schemas, execute in isolated subprocess. nanocode has `discover_tools()` but Aura's is more mature with proper isolation and schema generation.

### 4.3 Docker Sandbox

**Source:** `aura/sandbox.py`

**What:** Docker containers with `--memory=2g`, `--cpus=2`, `--pids-limit=200`, `--cap-drop=ALL`. Two modes: read-only rootfs (dynamic tools), read-write (terminal commands). nanocode has `sandbox/` — compare and adopt.

### 4.4 Vision Preprocessing

**Source:** `aura/vision.py`

**What:** Ollama vision model (`llama3.2-vision`) for screenshot descriptions. Plus native vision for providers that support it.

### 4.5 Research Sub-Agent (run_research)

**Source:** `aura/conversation/tools/web.py` (`run_research`)

**What:** Background sub-agent that generates search queries, searches web (Tavily), scrapes pages (BeautifulSoup), produces synthesized report.

### 4.6 Self-Updater

**Source:** `aura/updater.py`

**What:** Check GitHub releases, download ZIP, install via external updater script. Git checkout version uses fast-forward pull.

### 4.7 Continuation Reports

**Source:** `aura/conversation/manager.py` (ToolLimitState, continuation report XML format)

**What:** When tool limits are reached, generate an XML-structured continuation report so the model can continue without data loss.

### 4.8 Craft Compiler Pipeline

**Source:** `aura/craft/` (entire directory)

**What:** LLM output compilation pipeline that takes raw edits, compiles them into cleaner code, validates before user approval. Extremely sophisticated — low priority until the basics are solid.

## Phase 5: MiMo-Code Integrations

### Overview

MiMo-Code (`~/code/MiMo-Code`) is an open-source AI coding agent with advanced features that complement nanocode. These integrations focus on memory, task management, and context handling improvements.

**Why integrate from MiMo-Code?**
1. **Mature memory system**: SQLite FTS5 with BM25 ranking is production-tested
2. **Proven task management**: Tree-shaped tasks with lifecycle management
3. **Intelligent context**: Checkpoint-based reconstruction prevents context loss
4. **Self-improvement**: Dream/distill enables continuous learning
5. **Open source**: MIT license allows direct adoption

**MiMo-Code strengths:** Persistent memory with SQLite FTS5 search, tree-shaped task system, intelligent context reconstruction, skill discovery, dream/distill self-improvement, goal/stop conditions.

### 5.1 SQLite FTS5 Memory Search ✅

**Source:** `packages/opencode/src/memory/service.ts`

**What:** Add SQLite FTS5 full-text search with BM25 ranking for efficient memory retrieval. MiMo-Code uses:
- `memory_fts` table with FTS5 virtual table
- `bm25()` ranking function
- Score floor filtering (relative to top hit)
- Scope/type filtering

**Current nanocode gap:** Has SQLite storage but no FTS5 indexing for memory files.

**Implementation:**
1. ✅ Added FTS5 virtual table for memory content
2. ✅ Created memory indexer that parses MEMORY.md, checkpoint.md, notes.md
3. ✅ Implemented search with BM25 ranking
4. ✅ Added scope filtering (project, session, global)

**Effort:** ~4 hours

**Files created:**
- `nanocode/memory/__init__.py` - Module exports
- `nanocode/memory/indexer.py` - MemoryIndexer class with FTS5 indexing
- `nanocode/memory/search.py` - MemorySearch class with BM25 ranking
- `nanocode/memory/reconciler.py` - MemoryReconciler for keeping index in sync
- `tests/test_memory_fts.py` - Unit tests (13 tests passing)
- `scripts/test_memory_fts_simple.py` - Integration test script

### 5.2 Tree-shaped Task System ✅

**Source:** `packages/opencode/src/task/registry.ts`

**What:** Implement hierarchical task management with:
- Parent-child relationships (`T1`, `T1.1`, `T1.2`)
- Status lifecycle: `open` → `in_progress` → `blocked` → `done`/`abandoned`
- Event logging for all state changes
- Cleanup scheduling (configurable retention days)
- Session-aware task isolation

**Current nanocode gap:** Has `planning/` directory but minimal task tracking implementation.

**Implementation:**
1. ✅ Created TaskRegistry with SQLAlchemy models
2. ✅ Implemented task CRUD operations
3. ✅ Added status transition logic with validation
4. ✅ Created event logging system
5. ✅ Added cleanup/archiving based on retention policy

**Effort:** ~6 hours

**Files created:**
- `nanocode/planning/task_registry.py` - TaskRegistry class with full lifecycle management
- `nanocode/storage/models.py` - Added Task and TaskEvent models
- `tests/test_task_registry.py` - 12 unit tests (all passing)
