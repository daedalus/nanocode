# Plan: Port Features to nanocode from MiMo-Code + kilo

Sources:
- MiMo-Code: `~/code/MiMo-Code/packages/opencode/src/`
- kilo: `~/code/kilo/packages/opencode/src/`

Target: `/home/dclavijo/my_code/nanocode/nanocode/`

## Phase 1 — High-Impact, Self-Contained

### 1. Memory FTS Search
**Source**: MiMo-Code `memory/service.ts`, `memory/fts-query.ts`, `memory/fts.sql.ts`
**Target**: `nanocode/memory.py` (new module) or extend `storage/`

- SQLite FTS5 over memory markdown files
- `buildFtsQuery()`: tokenize user query → phrase-quoted OR-joined FTS5 MATCH
- BM25 ranking with relative score floor (0.15 of top hit)
- Scope/scope_id/type filtering
- Lazy reconcile before search (index on first query)

### 2. Batch Tool
**Source**: kilo `tool/batch.ts`, `tool/batch.txt`
**Target**: `nanocode/tools/batch.py` (new tool)

- Execute multiple tool calls in parallel via `asyncio.gather`
- Max 25 calls per batch, auto-truncate excess
- Disallow recursive batch calls
- Per-call status tracking (success/failure with error messages)
- Metadata summary: successful/failed counts, tool names
- **Why**: Nanocode currently executes tools sequentially. Batch is a 2-5x speedup for independent operations (read multiple files, grep + glob in parallel).

### 3. Commit Message Generator
**Source**: kilo `commit-message/generate.ts`, `commit-message/git-context.ts`, `commit-message/types.ts`
**Target**: `nanocode/git/commit_message.py` (new module)

- LLM-powered conventional commit message generation
- Git context: branch, recent commits (last 5), file statuses, diffs
- Lock file filtering (comprehensive list: package-lock, yarn.lock, poetry.lock, Cargo.lock, go.sum, etc.)
- Diff truncation at 4000 chars per file
- Conventional format: `type(scope): description` with body/footer
- Use small/fast model (configurable) for cost efficiency
- Retry with previous message context to avoid duplicates
- **Why**: nanocode has `/commit` but no LLM-generated messages. This is a direct UX win.

### 4. Code Review System
**Source**: kilo `kilocode/review/review.ts`, `kilocode/review/types.ts`, `kilocode/review/command.ts`
**Target**: `nanocode/git/review.py` (new module)

- Diff parser: unified diff → structured `DiffResult` with `DiffFile[]`, `DiffHunk[]`
- File status detection: added/modified/deleted/renamed
- Two review modes:
  - **Uncommitted**: `git diff HEAD` (staged + unstaged)
  - **Branch**: `git base...HEAD` (auto-detect base: main/master/dev/develop)
- Structured review prompt with severity levels:
  - CRITICAL (95%+): security, data loss, crashes
  - WARNING (85%+): bugs, logic errors, performance
  - SUGGESTION (75%+): code quality, best practices
- Output format: Summary → Issues table → Detailed findings → Recommendation
- Post-review action: offer to fix issues (mode switch to code/debug/orchestrator)
- **Why**: nanocode has no code review capability. This is a major feature addition.

## Phase 2 — Integration

### 5. Section-Aware Budgeted Reads
**Source**: MiMo-Code `session/budgeted-read.ts`
**Target**: `nanocode/context.py` or new `nanocode/budgeted.py`

- `parseSections()`: split markdown by `## ` headers
- `readBudgetedSectionAware()`: allocate token budget across sections, truncate within sections
- Fallback: header-only skeleton when budget is too small
- Truncation hint with offset for Read continuation

### 6. Structured Checkpoint Template
**Source**: MiMo-Code `session/checkpoint-templates.ts`, `session/checkpoint-paths.ts`
**Target**: `nanocode/checkpoint.py` (new module)

- 11-section template with per-section token budgets
- Sections: Active intent, Next action, Directives, Task tree, Current work, Files, Discovered knowledge, Errors, Live resources, Design decisions, Open notes
- `computeBoundary()`: token-budgeted role-aware boundary for tail preservation
- Path helpers: `metaDir()`, `checkpointPath()`, `memoryPath()`, `progressPath()`

### 7. Tool Output Pruning
**Source**: MiMo-Code `session/compaction.ts` (the `prune` function)
**Target**: `nanocode/context.py`

- Walk messages backwards from latest user turn
- After PRUNE_PROTECT tokens of tool calls, erase older tool_result content
- Protected tools (task, question, skill, memory) skip pruning
- Minimum prune threshold (PRUNE_MINIMUM = 20K tokens) to avoid trivial trims

### 8. Overflow Pressure Levels
**Source**: MiMo-Code `session/overflow.ts`
**Target**: `nanocode/context.py`

- `usable()`: compute usable context = context_limit - output_reserve - compaction_buffer
- `pressureLevel()`: 0 (<50%), 1 (<70%), 2 (<85%), 3 (≥85%)
- Replace nanocode's binary `is_overflow` check

### 9. Checkpoint Context Injection
**Source**: MiMo-Code `session/checkpoint.ts` (`renderRebuildContext`)
**Target**: `nanocode/core.py`

- Auto-inject checkpoint.md + MEMORY.md + notes.md into context after trim
- Budgeted section-aware reads for each file
- Task ledger with hierarchical subtask display
- "Already loaded" header to prevent agent from re-reading injected content

### 10. Task Progress Reconciliation
**Source**: MiMo-Code `session/checkpoint-progress-reconcile.ts`
**Target**: `nanocode/planning/task_registry.py`

- Per-task `progress.md` files written by subagents
- Checkpoint writer merges progress into task tree in checkpoint.md
- Subtask links with status icons

## Phase 3 — Session Management

### 11. Plan Follow-up (Mode Transition)
**Source**: kilo `kilocode/plan-followup.ts`
**Target**: `nanocode/core.py` or `nanocode/plan.py`

- After plan mode completes, ask user: "Start new session" or "Continue here"
- If new session: generate handover summary (discoveries, relevant files, implementation notes)
- Inject plan + handover + todo list into fresh session
- Auto-start implementation agent in new session
- **Why**: Complements checkpoint injection (Phase 2 #9) — kilo handles plan→code transition, MiMo-Code handles post-trim state restore.

### 12. Todo System (Complementary)
**Source**: kilo `session/todo.ts`, `tool/todo.ts`, `tool/todowrite.txt`, `tool/todoread.txt`
**Target**: `nanocode/planning/todo.py` (new module)

- `Todo.Info` schema: `{id, content, status, priority}`
- Statuses: `pending`, `in_progress`, `completed`, `cancelled`
- Priorities: `high`, `medium`, `low`
- `todowrite` tool: bulk-update entire list
- `todoread` tool: read current list
- Persist to storage (SQLite via existing `storage.py`)
- Bus events on update for UI reactivity
- **Why**: Nanocode's task system (`T1`, `T2`) is for tracked work items. Todo is a lightweight in-session checklist — different purpose, complementary.

### 13. Skill Discovery (Hierarchical)
**Source**: kilo `kilocode/paths.ts` (`skillDirectories`), `skill/discovery.ts`
**Target**: `nanocode/skills/discovery.py`

- Walk up directory tree from project root looking for `.nanocode/skills/*/SKILL.md`
- Check global `~/.nanocode/skills/` directory
- Priority order: project-local → parent dirs → global
- Return list of discovered skill paths
- **Why**: nanocode's skill system is flat. This adds hierarchical discovery matching common monorepo patterns.

## Phase 4 — Optional Enhancements

### 14. Max Mode (parallel best-of-N)
**Source**: MiMo-Code `session/max-mode.ts`
**Target**: `nanocode/llm/` (new module)

- Run N parallel completions
- Judge model selects best response
- Configurable via `experimental.maxMode`

### 15. Goal/Stop Condition
**Source**: MiMo-Code `session/goal.ts`
**Target**: `nanocode/core.py`

- `/goal` command sets a stopping condition
- Independent judge model evaluates whether condition is met
- Prevents premature optimistic stops

### 16. Scheduler
**Source**: kilo `scheduler/index.ts`
**Target**: `nanocode/scheduler.py` (new module)

- Simple interval-based task scheduler
- `register(task)`: id, interval_ms, run callback, scope (instance/global)
- Auto-cleanup on instance shutdown
- Timer-based with `.unref()` equivalent (non-blocking)
- **Why**: Useful for periodic memory reconciliation, cache cleanup, or watchdog tasks.

### 17. Config Injector
**Source**: kilo `kilocode/config-injector.ts`, `kilocode/rules-migrator.ts`, `kilocode/modes-migrator.ts`
**Target**: `nanocode/config.py` (extend existing)

- Migrate rules from `.nanocode/rules/` into system prompt
- Migrate custom modes/agents from `.nanocode/modes/` into config
- Merge permissions (read/edit patterns)
- Warning collection for skipped/invalid configs
- **Why**: nanocode already reads `.nanocode/` but lacks structured migration logic.

## Implementation Order

1. **Memory FTS** — biggest standalone improvement, no dependencies
2. **Batch Tool** — immediate speedup, no dependencies
3. **Commit Message** — direct UX win, self-contained
4. **Code Review** — major feature, requires diff parser
5. **Budgeted Reads** — needed by checkpoint injection (Phase 2)
6. **Checkpoint Template + Paths** — restructure checkpoint format
7. **Tool Pruning** — immediate context savings
8. **Pressure Levels** — better overflow detection
9. **Checkpoint Context Injection** — ties everything together
10. **Task Reconciliation** — requires checkpoint template changes
11. **Plan Follow-up** — improves plan→code workflow
12. **Todo System** — complements existing task system
13. **Skill Discovery** — hierarchical skill loading
14. **Max Mode** — optional, high effort
15. **Goal/Stop** — optional, medium effort
16. **Scheduler** — infrastructure for periodic tasks
17. **Config Injector** — config migration convenience
