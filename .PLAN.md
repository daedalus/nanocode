#### 1. Declarative Provider Profiles

**Source:** `providers/base.py`, `providers/__init__.py`, `plugins/model-providers/`

**What it is:** A `ProviderProfile` dataclass that declares everything about an inference provider — auth type, endpoints, temperature quirks, model catalog, per-provider hooks. Providers self-register via a `register_provider(profile)` call at import time. Auto-discovered from `plugins/model-providers/<name>/__init__.py` and `$HERMES_HOME/plugins/model-providers/` for user overrides.

**Key pattern:**
```python
@dataclass
class ProviderProfile:
    name: str
    api_mode: str = "chat_completions"  # or "anthropic", "bedrock", "codex"
    aliases: tuple = ()
    display_name: str = ""
    env_vars: tuple = ()                # env vars to look up for API key
    base_url: str = ""
    auth_type: str = "api_key"          # api_key|oauth_device_code|copilot|aws_sdk
    fixed_temperature: Any = None       # None=use default, OMIT_TEMPERATURE=don't send
    default_aux_model: str = ""         # cheap model for compression/vision
    fallback_models: tuple = ()         # curated model list for model picker

    def prepare_messages(self, messages) -> list[dict]: ...
    def build_extra_body(self, **context) -> dict: ...
    def build_api_kwargs_extras(self, **context) -> tuple: ...
    def fetch_models(self, api_key=None, timeout=8.0) -> list[str] | None: ...
```

**29 existing provider plugins** — Anthropic, OpenAI Codex, DeepSeek, Gemini, OpenRouter, Bedrock, Ollama Cloud, xAI, Kimi, Nous Portal, Alibaba/Qwen, MiniMax, NVIDIA NIM, Novita, StepFun, Xiaomi, Arcee, Azure Foundry, GMI Cloud, HuggingFace, KiloCode, Copilot, OpenCode Zen, ZAI, etc.

**Nanocode gap:** `core.py:438-582` (`_init_llm`) has hardcoded if/elif/else for each provider type. Adding a new provider means editing core.py.

**Porting effort:** ~400 lines of framework + ~100 lines per provider plugin. Replace `nanocode/llm/connectors/` structure.

**Porting strategy:**
```
nanocode/llm/
├── profiles/                      # NEW — replaces hardcoded dispatch
│   ├── __init__.py                # ProviderProfile dataclass + registry
│   └── providers/                 # auto-discovered plugins
│       ├── anthropic/__init__.py
│       ├── openai/__init__.py
│       ├── deepseek/__init__.py
│       └── ...
└── connectors/                    # keep transport implementations
    ├── openai.py
    ├── anthropic.py
    └── ollama.py
```

**Files to create:**
- `nanocode/llm/profiles/__init__.py` (port from `providers/base.py` + `providers/__init__.py`)

**Files to modify:**
- `nanocode/llm/router.py` — use profiles for config resolution
- `nanocode/core.py` — replace `_init_llm` if/else chain

#### 2. Transport Layer

**Source:** `agent/transports/` (6 files)

**What it is:** Abstract `ProviderTransport` that normalizes responses from different API shapes into a unified `NormalizedResponse`. Three implementations:
- `chat_completions.py` — OpenAI Chat Completions (covers 16+ providers)
- `anthropic.py` — Anthropic Messages API
- `codex.py` — OpenAI Codex/Responses API
- `bedrock.py` — AWS Bedrock

Each transport implements: `convert_messages()` (provider-format message adaptation), `convert_tools()`, `build_kwargs()`, `normalize_response()` (delta-based streaming → `NormalizedResponse`).

**Nanocode gap:** Message conversion and response normalization is inline in each connector. Adding a new API shape requires a new connector class.

**Porting effort:** ~800 lines. Clean abstraction that pays for itself with every new provider.

**Files to create:**
- `nanocode/llm/transports/__init__.py`
- `nanocode/llm/transports/base.py`
- `nanocode/llm/transports/chat_completions.py`
- `nanocode/llm/transports/anthropic.py`

**Files to modify:**
- `nanocode/llm/connectors/*.py` — refactor to use transports

#### 3. MCP Client with Sampling Support

**Source:** `tools/mcp_tool.py` (3593 lines)

**What it is:** Full MCP client supporting stdio/HTTP/SSE transports with:
- Auto-reconnection with exponential backoff
- Server sampling support (servers can request LLM completions — critical for advanced MCP server patterns)
- Parallel tool call opt-in
- Credential stripping from server env
- Connection lifecycle management

**Nanocode gap:** `mcp/__init__.py` (513 lines) has basic stdio/SSE. No sampling support, no reconnection, no parallel calls.

**Porting effort:** Selective port of sampling support and reconnection logic (~500 lines). Nanocode's MCP client structure is already reasonable.

**Files to modify:**
- `nanocode/mcp/__init__.py` — add sampling support, reconnection, parallel calls

#### 4. Lazy Dependency Installation

**Source:** `tools/lazy_deps.py` (616 lines)

**What it is:** Venv-scoped pip install at runtime for optional backends. Key design:
- Allowlist-based (`LAZY_DEPS` dict) — only known packages can be installed
- Venv-scoped only (`sys.executable`)
- Opt-out via `security.allow_lazy_installs: false`
- Security-conscious: no `--index-url`, `git+https://`, file paths
- Clear `FeatureUnavailable` exception with remediation hints

**Nanocode gap:** No lazy installs. Optional deps must be pre-installed.

**Porting effort:** ~600 lines, completely self-contained.

**Files to create:**
- `nanocode/tools/lazy_deps.py`

**Files to modify:**
- Backend tools to call `ensure("feature.name")` on first use

#### 5. Subagent Delegation

**Source:** `tools/delegate_tool.py`

**What it is:** Spawns child `AIAgent` instances with isolated context, restricted tool sets, own terminal sessions, and batch/parallel modes. The child agent gets its own config (tools disabled, different model, etc.) and runs in an isolated process.

**Nanocode gap:** `tools/task.py` exists but runs tasks within the same agent context. No isolated subagent architecture.

**Porting effort:** ~1000 lines. Depends on `AIAgent` instantiation pattern.

**Files to create:**
- `nanocode/agents/delegate.py` or `nanocode/tools/delegate.py`

**Files to modify:**
- `nanocode/core.py` — expose agent factory for re-use by delegate

#### 6. Self-Improvement Loop (Background Review + Curator)

**Source:** `agent/background_review.py`, `agent/curator.py`, `agent/curator_backup.py`

**What it is:** After every agent turn, forks a lightweight child agent call that reviews the conversation and decides if memory updates or skill creations are warranted. The curator periodically reviews agent-created skills for lifecycle transitions (pin/archive/consolidate).

**Nanocode gap:** Nanocode's skills system only discovers static files. No auto-creation, no lifecycle management.

**Porting effort:** ~600 lines. Background review is conceptually simple — one extra LLM call per turn with a structured prompt.

**Files to create:**
- `nanocode/agents/review.py` — background review logic
- `nanocode/agents/curator.py` — skill curator

**Files to modify:**
- `nanocode/core.py` — trigger review after each turn
- `nanocode/skills/__init__.py` — add lifecycle methods

## Porting Patterns

### Pattern 1: Direct Copy + Rename

When the hermes module is self-contained and nanocode has no equivalent:

```
hermes: tools/lazy_deps.py  →  nanocode: tools/lazy_deps.py
```

Adapt imports (`hermes_constants` → `nanocode.config`, etc.), rename internal references. Keep the logic identical.

### Pattern 2: Architecture Transplant

When nanocode has a stub or different structure for the same concept:

```
hermes: providers/base.py (ProviderProfile dataclass)
          + providers/__init__.py (registry + discovery)
          + plugins/model-providers/<name>/__init__.py (self-register)
       → 
nanocode: llm/profiles/base.py (same dataclass)
          + llm/profiles/registry.py (same registry + discovery)
          + llm/profiles/providers/<name>/__init__.py (self-register)
```

Then refactor existing connectors to use profiles instead of direct construction.

### Pattern 3: Selective Feature Extraction

When hermes has a massive file and nanocode only needs part:

```
hermes: tools/mcp_tool.py (3593 lines — full client)
       → 
nanocode: extract sampling support, reconnection logic
          inject into nanocode/mcp/__init__.py
```

Don't rewrite nanocode's MCP client — cherry-pick specific features.

### Pattern 4: Standalone Utility

When the module is a CLI pipeline, not library code:

```
hermes: trajectory_compressor.py (1508 lines)
       → 
nanocode: nanocode/trajectory/ (new package)
          entry point in pyproject.toml
```

## Key Architectural Differences

| Aspect | Hermes-Agent | Nanocode |
|--------|-------------|----------|
| Provider config | Declarative `ProviderProfile` plugins | Hardcoded if/else in `core.py:_init_llm` |
| Tool definition | Self-registering via `ToolRegistry` singleton | Same pattern |
| MCP client | 3593 lines with sampling + reconnection | 513 lines basic |
| Context compression | Structured summaries (Resolved/Pending/Remaining) | Simple summarization prompt |
| Subagents | Full isolated child agents with own terminal | `task.py` — same-process subtasks |
| Messaging | 15+ platform adapters with session management | 3 stub adapters |
| ACP | Full with tool mapping, approval, streaming | Basic protocol impl |
| LSP | 11 files | 1 file (629 lines, surprisingly complete) |
| Dependencies | Exact-pinned for supply-chain | Loose |
